Phase 2: กลยุทธ์แก้ปัญหา 8 ข้อแบบเจาะลึก (Solution Blueprint)  
นี่คือวิธีที่ผมจะเขียนโค้ดจัดการแต่ละปัญหา โดยใช้เครื่องมือที่มีอยู่แล้ว (Fuzzy Search, Levenshtein Distance) มาประยุกต์:  
กลุ่มที่ 1: ปัญหาเรื่อง "คน" (บุคคล)

Logic: เมื่อได้รับชื่อ "นาย สมชาย ใจดี" ให้ระบบค้นหาใน Master\_Persons ว่ามีชื่อไหนคล้ายกัน \>85% หรือไม่?  
ถ้ามี \-\> เสนอให้ผู้ใช้ยืนยัน "ใช่คนนี้หรือไม่?" \-\> ถ้าใช่ ให้ใช้ ID เดิม  
ถ้าไม่มี \-\> สร้างบันทึกใหม่

ข้อ 3 (LatLong ซ้ำ): เรื่องปกติของสถานที่ เช่น โกดังเดียวกัน ให้ใช้ Location\_ID เดียวกัน  
ข้อ 7 (คนชื่อเดียวกัน แต่ LatLong ไม่ตรง):

ข้อ 8 (คนละชื่อ แต่ LatLong ที่เดียวกัน):  
ระบบจะแนะนำ: "ตำแหน่งนี้เคยบันทึกในชื่อ 'บริษัท A' คุณต้องการใช้สถานที่เดียวกันหรือไม่?" เพื่อป้องกันการสร้างสถานที่ซ้ำซ้อน

ขั้นตอนที่ 2: ปรับปรุงกระบวนการรับข้อมูลใหม่ (Input Validation)  
แก้ไขฟังก์ชันใน Service\_SCG.gs และ WebApp.gs:  
เมื่อข้อมูลวิ่งเข้ามาจาก API \-\> ห้ามบันทึกลง Database ทันที  
ส่งเข้าฟังก์ชัน validateAndMatch()  
ระบบจะเช็คว่า:  
ชื่อนี้ มีใน Master ไหม? (Fuzzy Check)  
พิกัดนี้ มีใน Master ไหม? (Radius Check)  
ผลลัพธ์:  
Case A (มั่นใจ \>95%): บันทึกอัตโนมัติ โดยโยงหา ID ที่มีอยู่  
Case B (สงสัย): บันทึกลงในชีต Pending\_Review เพื่อให้ Admin กดอนุมัติก่อนลง Database จริง  
ขั้นตอนที่ 3: ตัดสิ่งที่ไม่จำเป็นออก (Cleanup)  
จากการวิเคราะห์โค้ดเดิม ส่วนที่แนะนำให้ ลบหรือปิดการใช้งาน ได้เลยหากไม่ใช้แล้ว:  
ฟังก์ชัน Debug เก่าๆ ที่เหลืออยู่ (เช่น testOldConnection)  
ชีตสำรองที่ไม่มีคนเข้าถึงเกิน 6 เดือน  
Trigger ที่ซ้ำซ้อน (ตรวจสอบใน Service\_AutoPilot.gs)  
💡 คำตอบคำถาม: "โค้ดอื่นๆ ผิดเหมือนกันมั้ย?"  
จากการตรวจสอบเบื้องต้น:  
ความถูกต้องของ Logic: โค้ดเดิม ไม่ผิด ในแง่การทำงาน (Functional) มันทำงานได้ตามที่คุณบอกในส่วนที่ 1  
จุดอ่อน: มันขาด "ชั้นของการตรวจสอบความถูกต้องของข้อมูล (Validation Layer)" ก่อนบันทึก จึงทำให้ข้อมูลสกปรกสะสมเรื่อยๆ  
สิ่งที่ต้องปรับ: ไม่ใช่การรื้อโค้ดทิ้ง แต่เป็นการ "แทรก" กระบวนการตรวจสอบ (Interceptor) เข้าไประหว่างขั้นตอน รับข้อมูล กับ บันทึกข้อมูล  
📝 สิ่งที่คุณควรทำต่อไป (Next Steps)  
ถ้าคุณพร้อมให้ผมเริ่ม ส่วนที่ 2 อย่างจริงจัง กรุณายืนยันคำสั่งต่อไปนี้:  
อนุญาตให้ผมสร้างไฟล์ใหม่:  
Service\_MasterData.gs (จัดการ Person/Location แบบแยกส่วน)  
Service\_DataCleansing.gs ( скрипт ทำความสะอาดข้อมูลเก่า)  
Index\_Review.html (หน้าเว็บสำหรับ Admin กดอนุมัติข้อมูลที่ระบบไม่แน่ใจ)  
ยืนยันโครงสร้างตารางใหม่: คุณโอเคกับแนวคิดการแยกตาราง Master\_Persons และ Master\_Locations หรือไม่?  
ตัวอย่างข้อมูล: หากคุณสามารถจำลองข้อมูลตัวอย่างสัก 10-20 แถว ที่มีปัญหาทั้ง 8 ข้อนี้ปนอยู่ ให้ผมเห็นภาพจริง จะช่วยให้ผมเขียน Algorithm ได้แม่นยำขึ้น 90% ครับ  
สรุปสั้นๆ: ระบบเดิมของคุณเหมือน "รถที่วิ่งเร็วแต่น้ำมันสกปรก" ส่วนที่ผมจะทำให้คือ "ติดตั้งระบบกรองน้ำมันชั้นดี" เพื่อให้รถวิ่งได้ไกลและเครื่องยนต์ไม่พังในระยะยาวครับ  
คุณต้องการให้ผมเริ่มร่างโค้ดของ Service\_MasterData.gs ให้ดูเป็นตัวอย่างก่อนเลยไหมครับ?
