### **📋 ส่วนที่ 1: วิเคราะห์โค้ดปัจจุบัน (Code Audit)**

**สิ่งที่ทำได้ดีมากแล้ว:**

1. **Schema Validation:** มีระบบตรวจโครงสร้างชีตก่อนทำงาน ป้องกัน Error จากการที่คนไปแก้มือ  
2. **Spatial Grid Indexing (O(N)):** ใน findHiddenDuplicates ใช้การจัดกลุ่มตามพื้นที่ (Grid) ก่อนคำนวณระยะทาง ช่วยให้ระบบไม่ค้างเมื่อข้อมูลเยอะ  
3. **Soft Delete & Merge Logic:** การใช้สถานะ Merged และ Inactive สำคัญมาก เพราะทำให้เราไม่เสียประวัติการส่งงานในอดีต  
4. **AI Tiered Resolution:** มีการแยกชั้นการทำงานของ AI (Normalization \-\> Retrieval \-\> AI Matching) ซึ่งเป็นแนวทางที่ถูกต้อง

**สิ่งที่ต้องปรับปรุงเพื่อให้แก้ปัญหา 8 ข้อได้:**

* **Context Missing:** AI ปัจจุบันมองแค่ "ชื่อ" แต่ปัญหาข้อ 5-8 จำเป็นต้องใช้ "จังหวัด/อำเภอ" มาเป็น Context ในการตัดสินใจ  
* **Conflict Resolution:** ระบบยังมีช่องโหว่ถ้าชื่อเหมือนกันแต่อยู่คนละที่ (Branch logic ยังไม่แข็งแรงพอ)

### **🎯 ส่วนที่ 2: การแก้ปัญหา 8 ข้อ (The 8 Commandments of Data Cleaning)**

ผมจะจัดกลุ่มปัญหา 8 ข้อของคุณออกเป็น 2 หมวดใหญ่ และวิธีแก้ดังนี้:

#### **หมวด A: ปัญหาด้าน "ชื่อ" (Identity) \- ข้อ 1, 4**

* **โจทย์:** ชื่อเขียนไม่เหมือนกัน, ชื่อซ้ำ, พิมพ์ผิด  
* **แนวทางแก้ไข:**  
  * **ใช้ NameMapping เป็นเกราะชั้นหน้า:** ทุกชื่อที่เข้ามา (Variant) ต้องถูก Map เข้าหา **Canonical UUID** (ตัวหลัก) เพียงหนึ่งเดียว  
  * **Deterministic Normalization:** ใน Utils\_Common.gs ฟังก์ชัน normalizeText ต้องเพิ่มการตัดคำที่เป็น Noise ออกให้หมด เช่น "จำกัด (มหาชน)", "สาขาที่...", "081-xxx" (ซึ่งคุณทำไว้บ้างแล้ว แต่ต้องเข้มงวดขึ้น)  
* 

#### **หมวด B: ปัญหาด้าน "สถานที่" (Location) \- ข้อ 2, 3, 5, 6, 7, 8**

* **โจทย์:** ชื่อเดียวกันแต่อยู่คนละที่, คนละชื่อแต่อยู่ที่เดียวกัน, พิกัดซ้ำ  
* **แนวทางแก้ไข:**  
  * **Geographic Uniqueness (Composite Key):** ระบบต้องไม่จำแค่ชื่ออย่างเดียว แต่ต้องจำแบบ **"ชื่อ \+ รหัสไปรษณีย์"** หรือ **"ชื่อ \+ อำเภอ"**  
  * **Spatial Buffer (50m Rule):** หากพิกัดห่างกันไม่เกิน 50 เมตร ให้สันนิษฐานว่าเป็น "จุดลงของเดียวกัน" (Drop Point) แม้ชื่อจะต่างกัน  
*

### **🚀 ถ้าโปรเจกต์นี้เป็นของผม... ผมจะทำอย่างไร? (Elite Architect Recommendation)**

หากผมเป็นคนคุมโปรเจกต์นี้ ผมจะเพิ่ม **"3 เลเยอร์แห่งความสะอาด"** ดังนี้ครับ:

#### **1\. เพิ่มระบบ "Composite Key" (ความปลอดภัยชั้นที่ 1\)**

แทนที่จะเช็คแค่ชื่อ ผมจะสร้างคีย์พิเศษหลังบ้าน: normalize(Name) \+ Postcode

* ถ้าคีย์นี้ตรงกัน \= **จบ** (Auto-match)  
* ถ้าชื่อตรงแต่ Postcode ต่าง \= **สงสัยว่าเป็นสาขาใหม่** (ส่งให้ AI/คน ตรวจ)

#### **2\. การทำความสะอาดแบบ "Cycle Clean" (ความปลอดภัยชั้นที่ 2\)**

ผมจะแก้โค้ดใน Service\_Master.gs ให้ทำงานเป็น Cycle:

1. **Dedupe LatLong:** ใครที่พิกัดซ้ำกันในระยะ 20 เมตร ต้องถูกยุบรวม (Merge) อัตโนมัติ โดยเอาชื่อที่พบบ่อยที่สุดเป็นชื่อหลัก  
2. **Reverse Geocoding Verification:** ทุกพิกัดที่ได้จากคนขับ ต้องรันผ่าน Google Maps API เพื่อดึง "ที่อยู่จริง" มาเทียบกับ "ที่อยู่ในใบงาน" ถ้าต่างกันเกินจังหวัด ให้แจ้งเตือน (Urgent Alert)

#### **3\. ระบบ "Human-in-the-loop" ผ่าน Web App (ความปลอดภัยชั้นที่ 3\)**

ข้อมูลที่ AI ไม่แน่ใจ (Confidence 70-85%) อย่าเพิ่งปล่อยเข้า Database

* สร้างหน้า **"Review Queue"** ใน Web App ให้แอดมินกด "Confirm Match" หรือ "Create New" ได้ง่ายๆ เพียงคลิกเดียว

### **🚀 แนวทางการสร้างฐานข้อมูลที่สะอาด (Elite Architect's Framework)**

ถ้าโปรเจกต์นี้เป็นของผม ผมจะเปลี่ยนจากระบบ "Database เดียว" เป็น **"Dual-Layer Core"** ดังนี้ครับ:

#### **เลเยอร์ที่ 1: Master Identity (จุดพิกัดหลัก)**

* **Key:** ใช้ LAT\_LNG เป็นตัวตั้ง (ปัดเศษทศนิยม 5 ตำแหน่งเพื่อความแม่นยำ 1 เมตร)  
* **หน้าที่:** เก็บข้อมูล "สถานที่" (Physical Location) 1 พิกัด \= 1 UUID เท่านั้น  
* **ตัวอย่าง:** พิกัดของห้างเซ็นทรัลชลบุรี จะมีเพียง 1 UUID แม้จะมีร้านข้างใน 10 ร้านมาส่งของก็ตาม

#### **เลเยอร์ที่ 2: Virtual Aliases (ชื่อเรียกขาน)**

* **หน้าที่:** เชื่อมโยงทุกชื่อที่พิมมา (Variant) เข้าหาพิกัดหลักใน Layer 1  
* **ตัวอย่าง:** "บีควิก", "B-Quik", "บีควิก สาขา..." ทั้งหมดนี้จะชี้ไปที่ UUID เดียวกันใน Layer 1

### **🛠️ วิธีแก้ปัญหา 8 ข้อแบบลงรายละเอียด (Step-by-Step)**

* **Algorithm: Static Name Stripping**

*   
* **AI Entity Linking:**  
  * แทนที่จะให้ AI หาแค่คำใกล้เคียง ให้ AI สรุป **"ประเภทร้าน"** เช่น AI เห็นชื่อ "จ.สะพรั่งภัณฑ์" ให้ Tag ว่า \[วัสดุก่อสร้าง\] เพื่อใช้แยกแยะ "คุณสมศักดิ์" ที่เป็นร้านค้า ออกจาก "คุณสมศักดิ์" ที่เป็นบ้านพัก (แก้ข้อ 1\)

#### **2\. การแก้เรื่องสถานที่และพิกัด (ข้อ 2, 3, 7, 8\)**

  * **ผลที่ได้:** แก้ข้อ 8 (คนละชื่อ ที่เดียวกัน) และข้อ 3 (LatLong ซ้ำ) ได้ 100%  
*   
* **Reverse Geocoding Conflict Check:**  
  * ปัจจุบันคุณมีคอลัมน์ PROVINCE และ DISTRICT  
  * ผมจะเพิ่มโค้ดตรวจสอบ: ถ้าชื่อคือ บี-ควิก แต่พิกัดที่คนขับส่งมาอยู่ที่ ระยอง ในขณะที่ Master ในระบบอยู่ที่ กรุงเทพ ระบบต้องแจ้งเตือนทันทีว่า **"นี่คือสาขาใหม่ ไม่ใช่สาขาเดิม"** (แก้ข้อ 6, 7\)

function normalizeText(text) {  
  if (\!text) return "";  
  var clean \= text.toString().toLowerCase();  
  // ลบ Job ID/Invoice Pattern (เช่น DN.12345, S260... P260...)  
  clean \= clean.replace(/(dn\\.\\d+|s\\d{9}|p\\d{9})/g, "");  
  // ลบ Noise อื่นๆ  
  var stopWords \= /บริษัท|บจก\\.?|บมจ\\.?|หจก\\.?|จำกัด|สาขาที่\\s?\\d+|โทร\\.?\\s?\\d+/g;  
  return clean.replace(stopWords, "").replace(/\[^a-z0-9\\u0E00-\\u0E7F\]/g, "").trim();  
}

**2\. สร้างฟังก์ชัน AutoMergeByProximity (หัวใจใหม่ของระบบ):**  
รันทุกคืนเพื่อกวาดฐานข้อมูล แถวไหนพิกัดใกล้กันเกิน 30-50 เมตร ให้ยุบรวมและย้ายชื่อไปไว้ใน NameMapping อัตโนมัติ

**3\. ยกระดับ AI Prompt ใน Service\_Agent.gs:**  
ส่ง SYS\_ADDR (ที่อยู่จากระบบ) ไปให้ AI ด้วย อย่าส่งแค่ชื่อ

* *Prompt ใหม่:* "จากชื่อ \[Name\] และที่อยู่ \[Addr\] นี้ เป็นร้านค้าสาขา หรือบ้านบุคคล? ถ้าเป็นร้านค้าสาขา ให้หา UUID ของสาขาหลักในจังหวัด \[Province\] มาให้ด้วย"

### **4\. วิเคราะห์แนวทางแก้ปัญหา 8 ข้อ (ถ้าโปรเจกต์นี้เป็นของผม)**

หากผมต้องคุมโปรเจกต์นี้ให้สะอาด 100% ผมจะทำตามขั้นตอน **"Triple Clean Strategy"** ดังนี้:

#### **ขั้นตอนที่ 1: การยุบพิกัด (The Geo-Anchor Rule)**

แก้ปัญหาข้อ 3, 5, 8 (พิกัดเดียวกันแต่คนละชื่อ)

* **แนวทาง:** ผมจะเปลี่ยนมาใช้พิกัดเป็นตัวตั้ง (Anchor) หากพิกัดใหม่ที่เข้ามาห่างจากจุดเดิมใน Database ไม่เกิน 20-30 เมตร ระบบจะ **"ห้ามเพิ่มแถวใหม่"** แต่จะบังคับให้ชื่อนั้นไปอยู่ใน NameMapping แทน  
* **ผลที่ได้:** Database จะมี 1 สถานที่ \= 1 UUID เท่านั้น ปัญหาพิกัดซ้ำจะหายไปทันที

#### **ขั้นตอนที่ 2: การแยกสาขาด้วย Context (The Branching Logic)**

แก้ปัญหาข้อ 6, 7 (ชื่อเดียวกันแต่อยู่คนละที่)

* **แนวทาง:** สร้างคีย์ผสม (Composite Key) คือ Normalized\_Name \+ Province  
* **ผลที่ได้:** "บีควิก" ในสมุทรปราการ กับ "บีควิก" ในชลบุรี จะถูกมองเป็นคนละ Entity ตั้งแต่แรก ระบบจะไม่อนุญาตให้เอาพิกัดชลบุรีไปทับพิกัดสมุทรปราการ

#### **ขั้นตอนที่ 3: การจัดการ Noise ในชื่อ (The Identity Strip)**

แก้ปัญหาข้อ 1, 2, 4 (ชื่อเขียนไม่เหมือนกัน/ชื่อซ้ำ)

* **แนวทาง:** เพิ่ม Regex ในฟังก์ชัน normalizeText เพื่อลบ Transaction ID (เช่น DN..., S26...) ทิ้งให้หมดก่อนประมวลผล  
* **ผลที่ได้:** คุณไก่-DN.123 และ คุณไก่-DN.456 จะถูกระบุว่าเป็น คุณไก่ (Entity เดียวกัน) ตั้งแต่ชั้น Normalization

#### **ขั้นตอนที่ 4: การให้ลำดับความสำคัญของข้อมูล (Data Authority)**

* **แนวทาง:** ข้อมูลที่คนขับอนุมัติ (Verified By Driver) จะมี Priority สูงสุดเหนือข้อมูลจากใบงานระบบ  
* **ผลที่ได้:** แก้ปัญหาข้อ 7 (ชื่อเดียวพิกัดคนละที่) โดยใช้จุดที่คนขับไปส่งจริงเป็นตัวตัดสิน (The Ground Truth)

**สรุปสิ่งที่ควรลบออก:**

1. **Checkbox ในชีต Database:** ถ้าข้อมูลหลักพันแถวขึ้นไป จะทำให้ไฟล์หนักมาก แนะนำให้ใช้การ Validation ด้วยคำพูด (Verified/Pending) หรือสีแทน  
2. **Logic "slice(0, 500)":** ในโค้ดมีการตัดข้อมูล 500 แถวมาตรวจ AI บ่อยครั้ง ควรเปลี่ยนเป็นระบบ **"Delta Processing"** คือตรวจเฉพาะแถวที่มีการเปลี่ยนแปลงล่าสุด (อ้างอิงคอลัมน์ UPDATED) เพื่อความเร็วครับ

คุณเห็นด้วยกับโครงสร้างและแนวทางนี้มั้ยครับ? ถ้าต้องการให้เริ่มเจาะจงที่ **Script ยุบรวมพิกัดซ้ำ (Geo-Merge)** แจ้งได้เลยครับ\!
