\# รายงานการวิเคราะห์ระบบ Logistics Master Data System (LMDS) และแนวทางการสร้างฐานข้อมูล (ส่วนที่ 2\)

\#\# 1\. สรุปโครงสร้างระบบปัจจุบัน (ภาพรวม 100%)

จากการวิเคราะห์โค้ดทั้งหมด 20 โมดูลใน GitHub Repository \`lmds\_4\_150426\` พบว่าระบบถูกออกแบบมาเป็น Google Apps Script (GAS) ที่ทำงานร่วมกับ Google Sheets โดยมีโครงสร้างดังนี้:

\#\#\# 1.1 โครงสร้างชีต (Google Sheets)  
ระบบมีการใช้งานชีตหลักทั้งหมด 14 ชีต ดังนี้:  
1\. \*\*Input\*\*: รับข้อมูล COOKIE และ ShipmentNos สำหรับดึงข้อมูลจาก SCG API  
2\. \*\*Data\*\*: เก็บข้อมูลงานประจำวันที่ดึงมาจาก SCG API (ผลลัพธ์จากส่วนที่ 1\)  
3\. \*\*ข้อมูลพนักงาน\*\*: เก็บข้อมูลพนักงานขับรถ (ID, ชื่อ, ทะเบียนรถ, Email, ฯลฯ)  
4\. \*\*สรุป\_Shipment\*\*: สรุปข้อมูลตาม Shipment  
5\. \*\*สรุป\_เจ้าของสินค้า\*\*: สรุปข้อมูลตามเจ้าของสินค้า  
6\. \*\*SCGนครหลวงJWDภูมิภาค\*\*: แหล่งข้อมูลดิบสำหรับส่วนที่ 2 (ข้อมูลที่คนขับส่งงานจริง)  
7\. \*\*Database\*\*: ฐานข้อมูลหลักเป้าหมาย (Master Data) ที่เก็บข้อมูลสถานที่และพิกัด  
8\. \*\*NameMapping\*\*: ชีตสำหรับจับคู่ชื่อที่เขียนต่างกันให้เป็น UID เดียวกัน (Alias/Variant)  
9\. \*\*GPS\_Queue\*\*: ระบบคิวสำหรับตรวจสอบพิกัดที่ผิดปกติจากคนขับ  
10\. \*\*PostalRef\*\*: ฐานข้อมูลรหัสไปรษณีย์และพิกัดระดับตำบล/อำเภอ  
11\. \*\*Blacklist\*\*: รายชื่อที่ถูกแบล็คลิสต์  
12\. \*\*SystemLogs\*\*: บันทึกการทำงานของระบบ  
13\. \*\*ErrorLogs\*\*: บันทึกข้อผิดพลาดของระบบ  
14\. \*\*Dashboard\*\*: แสดงสถิติและตัวชี้วัดต่างๆ

\#\#\# 1.2 โครงสร้างโมดูลโค้ด (20 ไฟล์ .gs และ .html)  
ระบบแบ่งการทำงานออกเป็นโมดูลต่างๆ อย่างชัดเจน:

1\. \*\*Config.gs\*\*: เก็บค่าคงที่ (Constants) และการตั้งค่าระบบทั้งหมด เช่น ชื่อชีต, Index ของคอลัมน์, การตั้งค่า AI (Gemini)  
2\. \*\*Menu.gs\*\*: จัดการเมนู UI บน Google Sheets (Custom Menu) สำหรับเรียกใช้งานฟังก์ชันต่างๆ  
3\. \*\*Service\_Master.gs\*\*: จัดการข้อมูลในชีต Database (Master Data) เช่น การสร้าง UUID, การคำนวณ Confidence/Quality Score, การทำ Deep Clean  
4\. \*\*Service\_SCG.gs\*\*: จัดการการดึงข้อมูลจาก SCG API (ส่วนที่ 1\) และนำข้อมูลมาใส่ในชีต Data รวมถึงการสร้างสรุป  
5\. \*\*Service\_GeoAddr.gs\*\*: จัดการการแปลงที่อยู่และพิกัด (Geocoding/Reverse Geocoding) ผ่าน Google Maps API  
6\. \*\*Utils\_Common.gs\*\*: ฟังก์ชันช่วยเหลือทั่วไป เช่น การคำนวณระยะทาง (Haversine), การแปลง Row เป็น Object, การ Normalize Text  
7\. \*\*Service\_AutoPilot.gs\*\*: ระบบอัตโนมัติ (Cron Job) สำหรับการทำงานเบื้องหลัง เช่น การทำ AI Indexing  
8\. \*\*WebApp.gs\*\*: จัดการ Web Application (HTML Service) สำหรับหน้าค้นหาข้อมูล  
9\. \*\*Service\_Search.gs\*\*: ระบบค้นหาข้อมูลใน Database โดยรองรับการค้นหาแบบ Fuzzy และ AI Keywords  
10\. \*\*Index.html\*\*: หน้าตา (UI) ของ Web Application สำหรับค้นหาข้อมูล  
11\. \*\*Setup\_Upgrade.gs\*\*: ฟังก์ชันสำหรับการอัปเกรดระบบ ตรวจสอบโครงสร้าง และหาข้อมูลซ้ำซ้อน (Hidden Duplicates)  
12\. \*\*Service\_Agent.gs\*\*: การทำงานของ AI Agent (Gemini) ในการวิเคราะห์ชื่อสถานที่และจับคู่ (Name Mapping)  
13\. \*\*Test\_AI.gs\*\*: ฟังก์ชันสำหรับทดสอบการทำงานของ AI และการเชื่อมต่อ Gemini API  
14\. \*\*Setup\_Security.gs\*\*: จัดการความปลอดภัย เช่น การตั้งค่า API Key และ Token สำหรับแจ้งเตือน  
15\. \*\*Service\_Maintenance.gs\*\*: การบำรุงรักษาระบบ เช่น การลบ Backup เก่า และตรวจสอบขนาดไฟล์ (Cell Limit)  
16\. \*\*Service\_Notify.gs\*\*: ระบบแจ้งเตือนผ่าน LINE Notify และ Telegram  
17\. \*\*Test\_Diagnostic.gs\*\*: ฟังก์ชันสำหรับตรวจสอบความสมบูรณ์ของระบบ (Dry Run)  
18\. \*\*Service\_GPSFeedback.gs\*\*: จัดการระบบ GPS Queue สำหรับตรวจสอบและอนุมัติพิกัดจากคนขับ  
19\. \*\*Service\_SchemaValidator.gs\*\*: ตรวจสอบโครงสร้างคอลัมน์ของชีตต่างๆ ว่าถูกต้องตาม Schema หรือไม่  
20\. \*\*Service\_SoftDelete.gs\*\*: จัดการสถานะของข้อมูล (Record Status) เช่น Active, Inactive, Merged

\#\# 2\. การวิเคราะห์ส่วนที่ 1 (ระบบปัจจุบัน)

จากที่คุณแจ้งว่า \*\*"ในส่วนที่ 1 นี้ ระบบทำงานดีอยู่แล้ว"\*\* ผมได้ตรวจสอบโค้ดใน \`Service\_SCG.gs\` และโมดูลที่เกี่ยวข้อง พบว่าระบบมีการทำงานที่สอดคล้องกับที่คุณอธิบายไว้ทุกประการ:

1\. \*\*การโหลดข้อมูล\*\*: แอดมินวาง Cookie และ ShipmentNos ในชีต \`Input\` แล้วกดปุ่ม ระบบจะเรียก \`fetchDataFromSCGJWD()\` เพื่อดึงข้อมูลจาก API  
2\. \*\*การจับคู่พิกัด\*\*: ระบบเรียก \`applyMasterCoordinatesToDailyJob()\` เพื่อนำชื่อปลายทางไปค้นหาใน \`Database\` และ \`NameMapping\` หากพบจะนำพิกัดมาใส่ในคอลัมน์ \`LatLong\_Actual\`  
3\. \*\*การดึง Email พนักงาน\*\*: ระบบมีการดึงข้อมูลจากชีต \`ข้อมูลพนักงาน\` มาใส่ในชีต \`Data\`  
4\. \*\*การสรุปข้อมูล\*\*: ระบบเรียก \`buildOwnerSummary()\` และ \`buildShipmentSummary()\` เพื่อสร้างสรุปในชีต \`สรุป\_เจ้าของสินค้า\` และ \`สรุป\_Shipment\`

\*\*ข้อสรุปสำหรับส่วนที่ 1\*\*: โค้ดส่วนนี้เขียนได้ดี มีการจัดการ Error และ Retry ที่เหมาะสม สามารถใช้งานต่อไปได้โดยไม่ต้องแก้ไขโครงสร้างหลัก

\#\# 3\. การออกแบบแนวทางระบบฐานข้อมูลใหม่ (ส่วนที่ 2\) และการแก้ปัญหา 8 ข้อ

เป้าหมายของส่วนที่ 2 คือการนำข้อมูลดิบจากชีต \`SCGนครหลวงJWDภูมิภาค\` (ข้อมูลที่คนขับส่งงานจริง) มาสร้างเป็นฐานข้อมูล (Master Data) ที่แข็งแกร่ง สะอาด และแก้ปัญหาความซ้ำซ้อนทั้ง 8 ข้อ

\#\#\# 3.1 โครงสร้างฐานข้อมูลเป้าหมาย (Target Database Architecture)

เพื่อให้ระบบสามารถแก้ปัญหาความซ้ำซ้อนได้ทั้งหมด เราต้องใช้สถาปัตยกรรมแบบ \*\*"Entity-Alias-Coordinate"\*\* ซึ่งในระบบปัจจุบันมีการปูพื้นฐานไว้แล้วผ่านชีต \`Database\` (Entity) และ \`NameMapping\` (Alias) แต่เราต้องปรับปรุงกระบวนการนำเข้าข้อมูล (Ingestion Pipeline) ให้รัดกุมขึ้น

\*\*โครงสร้างที่แนะนำ:\*\*  
1\. \*\*Master Entity (ชีต Database)\*\*: เก็บข้อมูลสถานที่ "1 สถานที่จริง \= 1 แถว (1 UUID)"  
2\. \*\*Alias Mapping (ชีต NameMapping)\*\*: เก็บชื่อเรียกที่แตกต่างกันของสถานที่เดียวกัน "หลายชื่อ \= 1 UUID"  
3\. \*\*Coordinate History (ชีต GPS\_Queue หรือสร้างใหม่เป็น GPS\_History)\*\*: เก็บประวัติพิกัดที่คนขับส่งมา เพื่อหาค่าเฉลี่ยพิกัดที่แม่นยำที่สุด

\#\#\# 3.2 แนวทางการแก้ปัญหา 8 ข้อ

\#\#\#\# ปัญหาที่ 1: เรื่องชื่อบุคคลซ้ำกัน (ชื่อเหมือนกันเป๊ะ)  
\* \*\*ปัญหา\*\*: มีชื่อ "ร้าน ก." เข้ามาหลายครั้งในวันต่างกัน  
\* \*\*แนวทางแก้ไข\*\*:   
  \* ใช้ฟังก์ชัน \`normalizeText()\` เพื่อลบช่องว่างและอักขระพิเศษ  
  \* เมื่อมีข้อมูลใหม่เข้ามา ระบบจะค้นหาชื่อนี้ใน \`Database\` ก่อน หากพบชื่อตรงกันเป๊ะ (Exact Match) ระบบจะไม่สร้างแถวใหม่ แต่จะนำพิกัดใหม่ไปเปรียบเทียบกับพิกัดเดิม (ดูปัญหาที่ 7\)

\#\#\#\# ปัญหาที่ 2: เรื่องชื่อสถานที่อยู่ซ้ำ (ที่อยู่เหมือนกันเป๊ะ)  
\* \*\*ปัญหา\*\*: มีที่อยู่ "123 ม.1 ต.ก กทม." เข้ามาหลายครั้ง  
\* \*\*แนวทางแก้ไข\*\*:  
  \* สร้างคอลัมน์ \`Address\_Hash\` (การเข้ารหัสที่อยู่) หรือใช้ \`normalizeText()\` กับที่อยู่  
  \* หากชื่อต่างกันแต่ที่อยู่เหมือนกันเป๊ะ ระบบจะแจ้งเตือน (Flag) ให้แอดมินตรวจสอบว่าเป็นสถานที่เดียวกันหรือไม่ (ดูปัญหาที่ 5\)

\#\#\#\# ปัญหาที่ 3: เรื่อง LatLong ซ้ำกัน (พิกัดเหมือนกันเป๊ะ)  
\* \*\*ปัญหา\*\*: มีพิกัด "13.1234, 100.1234" เข้ามาหลายครั้ง  
\* \*\*แนวทางแก้ไข\*\*:  
  \* ใช้ Spatial Grid Algorithm (ที่มีอยู่ใน \`Setup\_Upgrade.gs\`) เพื่อจัดกลุ่มพิกัดที่อยู่ใกล้กัน (เช่น รัศมี 50 เมตร)  
  \* หากพิกัดซ้ำกันเป๊ะ ระบบจะตรวจสอบชื่อ หากชื่อต่างกัน จะเข้าสู่กระบวนการตรวจสอบ (ดูปัญหาที่ 8\)

\#\#\#\# ปัญหาที่ 4: เรื่องบุคคลเดียวกันแต่ชื่อเขียนไม่เหมือนกัน (หลายบริษัทพิมพ์ชื่อต่างกัน)  
\* \*\*ปัญหา\*\*: "โลตัส บางนา", "Lotus Bangna", "บ.เอก-ชัย (สาขาบางนา)" คือที่เดียวกัน  
\* \*\*แนวทางแก้ไข\*\*:  
  \* ใช้ชีต \`NameMapping\` เป็นตัวกลาง  
  \* ใช้ AI Agent (Gemini) ที่มีอยู่ใน \`Service\_Agent.gs\` เพื่อวิเคราะห์ความคล้ายคลึงของชื่อ (Fuzzy Matching & Semantic Matching)  
  \* หาก AI มั่นใจสูง (Confidence \> 90%) ให้จับคู่ (Auto-Map) เข้ากับ UUID เดียวกันอัตโนมัติ  
  \* หากมั่นใจปานกลาง (70-89%) ให้ส่งเข้าคิวให้แอดมินตรวจสอบ (Review Queue)

\#\#\#\# ปัญหาที่ 5: เรื่องบุคคลคนละชื่อ แต่ชื่อสถานที่อยู่เดียวกัน  
\* \*\*ปัญหา\*\*: "ร้านสมชาย" และ "บจก. สมชายการค้า" อยู่ที่ "123 ม.1..." เหมือนกัน  
\* \*\*แนวทางแก้ไข\*\*:  
  \* เมื่อระบบพบที่อยู่ซ้ำกัน (จากปัญหาที่ 2\) ระบบจะแนะนำให้แอดมินทำการ \*\*Merge (รวมข้อมูล)\*\*  
  \* ใช้ฟังก์ชัน \`mergeUUIDs()\` ใน \`Service\_SoftDelete.gs\` เพื่อยุบรวมให้เหลือ Master UUID เดียว และนำชื่อที่เหลือไปใส่ใน \`NameMapping\`

\#\#\#\# ปัญหาที่ 6: เรื่องบุคคลชื่อเดียวกัน แต่ชื่อสถานที่อยู่ไม่เหมือนกัน  
\* \*\*ปัญหา\*\*: "ร้าน ก." อยู่ที่ "กทม." และ "ร้าน ก." อยู่ที่ "เชียงใหม่" (อาจเป็นสาขา หรือคนละร้าน)  
\* \*\*แนวทางแก้ไข\*\*:  
  \* ระบบต้องไม่ใช้ "ชื่อ" เป็น Primary Key อย่างเดียว แต่ต้องใช้ "ชื่อ \+ พิกัด (หรือจังหวัด)" เป็น Composite Key  
  \* หากชื่อเหมือนกัน แต่พิกัดห่างกันเกิน 5 กิโลเมตร (คำนวณด้วย Haversine) ระบบจะถือว่าเป็น \*\*"คนละสถานที่ (สาขา)"\*\* และสร้าง UUID ใหม่แยกกัน

\#\#\#\# ปัญหาที่ 7: เรื่องบุคคลชื่อเดียวกัน แต่เลข Lat Long คนละที่ ไม่เหมือนกัน  
\* \*\*ปัญหา\*\*: "ร้าน ก." วันนี้คนขับส่งพิกัด A พรุ่งนี้คนขับอีกคนส่งพิกัด B (ห่างกัน 200 เมตร)  
\* \*\*แนวทางแก้ไข\*\*:  
  \* ใช้ระบบ \`GPS\_Queue\` ที่มีอยู่แล้ว  
  \* หากพิกัดใหม่ห่างจากพิกัดใน Database เกิน Threshold (เช่น 50 เมตร) ระบบจะนำพิกัดใหม่เข้า \`GPS\_Queue\`  
  \* แอดมินสามารถดูแผนที่และกด Approve เพื่ออัปเดตพิกัดใน Database หรือ Reject หากพิกัดใหม่ผิด

\#\#\#\# ปัญหาที่ 8: เรื่องบุคคล คนละชื่อ แต่ เลข Lat Long ที่เดียวกัน  
\* \*\*ปัญหา\*\*: "ร้าน ก." และ "ร้าน ข." มีพิกัดเดียวกันเป๊ะ (อาจเป็นตึกเดียวกัน หรือคนขับกดส่งงานผิดที่)  
\* \*\*แนวทางแก้ไข\*\*:  
  \* ใช้ Spatial Grid Algorithm ตรวจสอบพิกัดที่ซ้ำซ้อน (Hidden Duplicates)  
  \* ระบบจะแจ้งเตือน (Flag) ว่ามี 2 ชื่อในพิกัดเดียวกัน  
  \* แอดมินตรวจสอบ: หากเป็นร้านเดียวกัน ให้ทำ Merge (ปัญหาที่ 5\) หากเป็นคนละร้าน (เช่น ห้างสรรพสินค้า) ให้เก็บแยก UUID ไว้ตามปกติ

\#\#\# 3.3 กระบวนการทำงาน (Workflow) สำหรับส่วนที่ 2

เพื่อให้ระบบฐานข้อมูลสะอาดและใช้งานได้จริง ผมขอเสนอ Workflow การนำเข้าข้อมูลจากชีต \`SCGนครหลวงJWDภูมิภาค\` ดังนี้:

\*\*ขั้นตอนที่ 1: Data Extraction & Pre-processing (ดึงและทำความสะอาดเบื้องต้น)\*\*  
\* ดึงข้อมูลใหม่จากชีต \`SCGนครหลวงJWDภูมิภาค\` (เฉพาะแถวที่ \`SYNC\_STATUS\` ยังไม่เป็น "SYNCED")  
\* ทำความสะอาดชื่อ: ลบช่องว่าง, แปลงตัวพิมพ์เล็ก/ใหญ่, ตัดคำนำหน้าทั่วไป (บจก., ร้าน) ด้วย \`normalizeText()\`  
\* ตรวจสอบความถูกต้องของพิกัด (Lat/Long ต้องเป็นตัวเลขและอยู่ในประเทศไทย)

\*\*ขั้นตอนที่ 2: Entity Resolution (การค้นหาและจับคู่)\*\*  
\* นำชื่อที่ทำความสะอาดแล้วไปค้นหาใน \`Database\` และ \`NameMapping\`  
\* \*\*กรณีที่ 1 (Exact Match)\*\*: พบชื่อตรงกันเป๊ะ \-\> ข้ามไปขั้นตอนที่ 3  
\* \*\*กรณีที่ 2 (Fuzzy/AI Match)\*\*: ไม่พบชื่อตรงเป๊ะ ส่งให้ AI วิเคราะห์  
  \* ถ้า AI จับคู่ได้ (Confidence \> 90%) \-\> บันทึกลง \`NameMapping\` และข้ามไปขั้นตอนที่ 3  
  \* ถ้า AI ไม่แน่ใจ \-\> สร้างเป็น Entity ใหม่ (UUID ใหม่) ใน \`Database\`

\*\*ขั้นตอนที่ 3: Coordinate Validation (การตรวจสอบพิกัด)\*\*  
\* นำพิกัดจากคนขับ (LatLong\_Driver) มาเทียบกับพิกัดในฐานข้อมูล (LatLong\_DB)  
\* คำนวณระยะทางด้วย Haversine Formula  
\* ถ้าระยะทาง \< 50 เมตร \-\> ถือว่าปกติ (อาจนำพิกัดมาหาค่าเฉลี่ยเพื่อความแม่นยำขึ้น)  
\* ถ้าระยะทาง \> 50 เมตร \-\> ส่งข้อมูลเข้าชีต \`GPS\_Queue\` เพื่อให้แอดมินตรวจสอบ

\*\*ขั้นตอนที่ 4: Post-processing & Sync\*\*  
\* อัปเดต \`SYNC\_STATUS\` ในชีต \`SCGนครหลวงJWDภูมิภาค\` เป็น "SYNCED"  
\* คำนวณ Quality Score และ Confidence Score ใหม่สำหรับข้อมูลที่มีการอัปเดต

\#\# 4\. บทสรุปและคำแนะนำเพิ่มเติม

หากโปรเจกต์นี้เป็นของผม ผมจะดำเนินการดังนี้เพื่อให้ระบบสมบูรณ์ที่สุด:

1\. \*\*เปิดใช้งาน AI Agent อย่างเต็มรูปแบบ\*\*: โค้ดใน \`Service\_Agent.gs\` มีการเขียน Prompt สำหรับ Gemini ไว้ดีมากแล้ว ควรตั้ง Trigger ให้ AI ทำงานทุกคืนเพื่อจัดกลุ่มชื่อที่พิมพ์ต่างกัน (Name Mapping) อัตโนมัติ ซึ่งจะแก้ปัญหาข้อ 4 ได้อย่างเด็ดขาด  
2\. \*\*ใช้ระบบ GPS Queue เป็นหัวใจหลัก\*\*: ปัญหาพิกัดเพี้ยน (ข้อ 7\) เป็นปัญหาคลาสสิกของระบบ Logistics การให้แอดมินตรวจสอบผ่าน \`GPS\_Queue\` ก่อนอัปเดต Master Data เป็นวิธีที่ปลอดภัยที่สุด  
3\. \*\*การทำความสะอาดข้อมูลย้อนหลัง (Data Cleansing)\*\*: ก่อนเริ่มใช้ระบบใหม่ ควรนำข้อมูลเก่าใน \`Database\` มารันผ่านฟังก์ชัน \`findHiddenDuplicates()\` ใน \`Setup\_Upgrade.gs\` เพื่อหาพิกัดที่ทับซ้อนกัน (ข้อ 8\) และทำการ Merge ให้เรียบร้อย  
4\. \*\*สิ่งที่ไม่จำเป็นและสามารถลบได้\*\*:  
   \* โค้ดส่วนใหญ่ถูกเขียนมาแบบ Modular และมีการใช้งานจริง อย่างไรก็ตาม หากคุณไม่ได้ใช้ Telegram ในการแจ้งเตือน สามารถปิดการทำงานส่วน Telegram ใน \`Service\_Notify.gs\` ได้เพื่อลดความซ้ำซ้อน  
   \* ชีต \`Blacklist\` หากไม่ได้ใช้งานจริงในการกรองข้อมูล สามารถลบออกได้เพื่อลดความซับซ้อนของระบบ

ระบบนี้มีโครงสร้างพื้นฐาน (Architecture) ที่แข็งแกร่งมากระดับ Enterprise (มีการทำ Caching, Soft Delete, Schema Validation) หากปรับปรุง Workflow การนำเข้าข้อมูลตามที่เสนอไป จะสามารถแก้ปัญหาทั้ง 8 ข้อได้อย่างยั่งยืนครับ
